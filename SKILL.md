# Iteration Planning

把零散改动整理成结构化清单，按依赖顺序逐项执行。

## 适用场景

- 用户需求已清晰（知道要改什么）
- 改动琐碎但数量多
- 需要有条理、不遗漏、有顺序的执行

**不适用：**
- 模糊需求 → 用 brainstorming
- 单一简单改动 → 直接对话处理

---

## 流程

### Step 1: 收集改动

让用户陈述改动内容，可以口语化、碎片化。

如果用户描述不够具体，追问：
- 改动涉及哪些文件/函数？
- 改动目的是什么？（方便判断影响范围）

**不要追问"你真正想做什么"** — 用户需求已清晰，只需要整理细节。

### Step 2: 整理清单

把用户描述转换成结构化清单：

```markdown
## 改动清单

### 模块A（依赖：无）
| # | 改动项 | 类型 | 位置 | 状态 |
|---|--------|------|------|------|
| 1 | ... | 新增/修改/删除 | 文件路径:函数名 | 待执行 |
| 2 | ... | ... | ... | 待执行 |

### 模块B（依赖：模块A完成）
| # | 改动项 | 类型 | 位置 | 状态 |
|---|--------|------|------|------|
| 3 | ... | ... | 文件路径:行号 | 待执行 |
```

**分类维度：**
- 默认：按模块分组 + 按执行顺序排序（依赖关系）
- 用户可指定其他维度

**位置标注格式：**
- 函数级：`src/auth/login.ts:verifyToken()`
- 行级：`src/auth/login.ts:第50行`
- 文件级：`src/auth/login.ts（新增文件）`

### Step 3: 排序

按依赖关系确定执行顺序：

1. 先标注每个模块的依赖项
2. 无依赖的模块排前面
3. 依赖其他模块的排后面
4. 同模块内按调用链排序（底层→上层）

**依赖标注格式：**
- `依赖：无` — 可立即执行
- `依赖：模块A完成` — 等待前置模块
- `依赖：改动项#1完成` — 等待特定改动项

### Step 4: 确认

向用户展示清单，询问：

```
清单已整理完毕，共 N 项改动。
请核对：
1. 有遗漏的改动项吗？
2. 执行顺序正确吗？
3. 模块分组合理吗？

确认后我开始逐项执行。
```

等待用户确认。用户可能：
- 补充遗漏项 → 更新清单，再次确认
- 调整顺序 → 重新排序
- 确认无误 → 进入执行阶段

### Step 5: 执行

**逐项确认模式（默认）：**

每次执行前：
```
即将执行 #1: [改动项描述]
位置: [文件:函数]
类型: [新增/修改/删除]

确认执行？
```

用户确认后执行，完成后更新状态：
- `待执行` → `已完成`
- 如遇阻塞 → 标记 `阻塞: [原因]`，询问用户处理方式

每项完成后汇报：
```
#1 已完成 ✓
下一项: #2 [改动项描述]
继续？
```

## 输出格式

### 清单模板

```markdown
# 改动清单

**项目：** [项目名称]
**日期：** [YYYY-MM-DD]
**总项数：** N
**状态：** 执行中 / 已完成 / 部分完成

---

## 模块分组

### [模块名]（依赖：[依赖项]）

| # | 改动项 | 类型 | 位置 | 状态 |
|---|--------|------|------|------|
| 1 | 具体改动描述 | 新增 | path:func | 已完成 |
| 2 | ... | 修改 | path:line | 待执行 |

---

## 执行记录

- **#1** [时间] 已完成 — [简要说明]
- **#2** [时间] 待执行
```

### 完成汇报模板

```markdown
## 执行完成汇报

**已完成：** N/N 项
**耗时：** X 分钟

### 模块A
- ✅ #1 [改动项] — [位置]
- ✅ #2 [改动项] — [位置]

### 模块B
- ✅ #3 [改动项] — [位置]

### 遗留事项
- [如有阻塞项，列出处理建议]
```

---

## 注意事项

1. **不要添加未被要求的改动** — 只做用户清单里的内容
2. **遇到不确定的细节先问** — 位置、参数、边界条件
3. **保持清单可追溯** — 每项改动都能追溯到用户原始描述
4. **阻塞时立即汇报** — 不要自行跳过或绕过
